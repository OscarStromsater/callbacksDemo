Callbacks






Callbacks in the wild

One of the ways to know that you are encountering a callback
is that you will see a function being passed as a parameter to another function.
a great example of this is the addeventlistener which will be demonstrated below.
An EventListener takes two parameters, what interaction it is
listening for i.e. a 'click' and the second is the callback. The callback tells the listener what function to run when the event happens.

The example above shows an example of how to use a callback asynchronously. The eventlistener is initialised but the callback does not trigger until later and doesn't block
the event loop.

The example above shows an example of how to use a callback synchronously.

Caveats

The biggest caveat when working with callbacks is that they are not inherently Asynchronous, they can also be used synchronously.
Another caveat is that you should pass the callback as the last parameter to a function.

Pros and cons

Pros
  • Allows you to access data which would normally be out of scope
  • Allows for interactivity to webpages in conjucture with addeventlistener
  • Allows for nonblocking code to be run
  • Using setTimout allows for non nested callbacks <=(this is a joke)
Cons
  • Not inherently Asynchronous
  • Very easy to make code unreadable when many nested callbacks are used
  • Callback Hell



  • Troublesome to trace
  • Wrapping our head around the scoping of the functions and
    pushing data to variables declared outside the functions
  • Following the data flow between the callbacks
  • Callback Hell!
